# **Manga Creation & Publishing Management System**

## **Top 50 High-Impact Business Rules — Filtered from 136 BR Catalog**

**Selection Criteria:** Rules were selected based on (1) all 40 Critical-level BRs were considered first; (2) High-level BRs that form essential gates in the main flow or underpin Critical rules were selected next; (3) redundant or derivative rules were excluded. Highlighted rows (**red**) \= Critical; white rows \= High. Total: 31 Critical \+ 19 High.

| **P1 — Series Proposal** |  |  |  |  |  |

The gate preventing incomplete proposals from entering the board voting pipeline. |
| **BR-24** | **Activation Preconditions** | *Behavioral* | **Critical** | P1 / P2 | A series can only become Active when a valid BoardDecision with quorum has been finalized AND is reviewed by a Tantou Editor. | **BR-18** | Publication Type Governance | *Behavioral* | **High** | P1 — Series Proposal | Publication type must be chosen from a system-configured list. After a series is Active, any change requires a BoardDecision. Ensures no silent reclassification of a series without board oversight. |
| **BR-19** | Single Active Proposal Limit | *Behavioral* | **High** | P1 — Series Proposal | A Mangaka may have at most 1 proposal in Pending or Under Review at any time. Prevents flooding the editorial board with proposals from a single author and keeps review load manageable. |
| **BR-21** | Series Lifecycle Policy | *Structural* | **High** | P1 — Series Proposal | Defines the canonical series states: Draft → Under Review → Approved → Active → On-Hold | Cancelled. Cancelled series cannot be reactivated through normal UI, enforcing finality of cancellation decisions. |
| **P2 — Editorial Board Voting** |  |  |  |  |  |
| **BR-27** | **Voting Eligibility** | *Behavioral* | **Critical** | P2 — Board Voting | Only active Editorial Board members with no conflict of interest may vote on any BoardDecision. The primary fairness gate for all publication decisions. |
| **BR-28** | **Conflict of Interest Definition** | *Definitional* | **Critical** | P2 — Board Voting | Defines conflict of interest as: being the series' Mangaka, assigned Tantou Editor, assigned Assistant, or the creator of the proposal/decision. The system must exclude these votes in real time. |
| **BR-29** | **Quorum Requirement** | *Structural* | **Critical** | P2 — Board Voting | Any BoardDecision requires a minimum of 3 valid (conflict-free) votes to produce a result. Without quorum the decision cannot be finalized. Applies equally to new series approvals and cancellation decisions. |
| **BR-30** | **Invalid Vote Exclusion** | *Behavioral* | **Critical** | P2 — Board Voting | Votes cast by members with a conflict of interest do not count toward quorum or the majority calculation. The system must validate eligibility before recording each vote to prevent retroactive disputes. |
| **BR-33** | **Majority Rule** | *Calculational* | **Critical** | P2 — Board Voting | A decision is Approved when Approve votes \> 50% of valid votes. If Approve and Reject votes are equal, the system must raise a notification to the Editor-in-Chief for manual resolution. This defines the exact formula for auto-finalization while escalating tied decisions for editorial oversight.  |
| **BR-34** | **Finalization Rule** | *Behavioral* | **Critical** | P2 — Board Voting | A decision is finalized only when: (quorum met AND voting window expired) OR the current votes make the outcome mathematically irreversible. Prevents premature finalization and ensures due process. |
| **BR-31** | Voting Window | *Temporal* | **High** | P2 — Board Voting | Every BoardDecision must have a deadline. For proposal reviews the default voting window is 7 calendar days from assignment. Ensures the board does not stall indefinitely on any decision. |
| **BR-35** | Reject Reason Requirement | *Behavioral* | **High** | P2 — Board Voting | A Reject vote must include a reason of at least 50 characters. Ensures Mangaka receives actionable feedback and provides an audit trail for rejection patterns. |
| **BR-37** | Deferred & Expired Handling | *Behavioral* | **High** | P2 — Board Voting | If a decision has no final result when the voting window closes, it moves to Expired or Deferred and the Editor-in-chief is notified. Prevents decisions from silently stalling and blocking series activation. |
| **P3 — Chapter Creation & Task Assignment** |  |  |  |  |  |

| **BR-46** | **Ready-for-Submission Condition** | *Behavioral* | **Critical** | P3 / P4 | A chapter can only advance to Submitted when all required PageTasks are Approved. The bridge between page-level progress and manuscript submission — the single most impactful gate in the P3→P5 flow. |
| **BR-49** | **Page Range Non-Overlap** | *Structural* | **Critical** | P3 — Task Assignment | Active PageTask page ranges within the same chapter must not overlap. Overlapping ranges cause two assistants to work on the same pages, leading to conflicting submissions and wasted effort. |
| **BR-54** | **Due Date Validation** | *Temporal* | **Critical** | P3 — Task Assignment | PageTask dueDate must not predate creation date, must be ≤ chapter deadline, and must be at least 3 days before the chapter submission deadline. Ensures assistants have a realistic working window. |
| **BR-57** | **Assistant-Only Assignment** | *Behavioral* | **Critical** | P3 — Task Assignment | Tasks may only be assigned to users with an active Assistant role. Prevents accidental assignment to Editors or Board members who have no task-execution obligation in the system. |
| **BR-41** | Chapter Numbering Policy | *Structural* | **High** | P3 — Chapter Creation | Chapter numbers must be unique within a series and increase monotonically. Special/Bonus chapters use a separate chapterType. Prevents ordering ambiguity in the editorial review and publishing schedule. |
| **BR-43** | Deadline Escalation | *Temporal* | **High** | P3 — Chapter Creation | When a chapter becomes overdue the system must alert the Tantou Editor within 1 hour of detection. Early escalation gives editors time to intervene before publication is jeopardized. |
| **BR-52** | Assignment Authority | *Behavioral* | **High** | P3 — Task Assignment | Only the Mangaka owner may assign PageTasks; assistants cannot self-assign. Preserves the Mangaka's editorial control over their chapter workflow. |
| **BR-59** | Mandatory Task Fields | *Structural* | **High** | P3 — Task Assignment | A PageTask must have pageRange, taskType, dueDate, and assistantId. Missing any field blocks task creation. Prevents skeleton tasks that generate incomplete progress calculations. |
| **P4 — Page-Level Progress Tracking** |  |  |  |  |  |
| **BR-60** | **Task Completion Rule** | *Behavioral* | **Critical** | P4 — Progress Tracking | A PageTask is completed only when the Mangaka marks it Approved. Submitted alone does not count. This rule (BR-60) underpins the chapter completion formula and the manuscript submission gate (BR-67). |
| **BR-67** | **Manuscript Submission Requirement** | *Behavioral* | **Critical** | P4 / P5 | Mangaka may only submit a manuscript when all required PageTasks are Approved. The culmination of the page-tracking pipeline — without this gate a manuscript could be submitted with incomplete artwork. |
| **BR-69** | **Assistant Status Transition Restriction** | *Behavioral* | **Critical** | P4 — Progress Tracking | Assistants may only transition tasks Pending→In-Progress and In-Progress→Submitted. They cannot Approve or Reject their own work. Separates execution from quality control. |
| **BR-61** | Chapter Completion Formula | *Calculational* | **High** | P4 — Progress Tracking | Chapter completion % \= Approved Tasks / Total Required Tasks × 100%. The metric displayed to editors for at-a-glance chapter health and required by BR-84 to allow manuscript approval. |

| **BR-74** | **Editor Access Restriction** | *Behavioral* | **Critical** | P6 — Editor Review | Only the Tantou Editor assigned to a series may review its manuscripts. Without this control an editor from a rival series could access or approve another's work. |

| **BR-73** | Versioning Policy | *Behavioral* | **High** | P5 — Manuscript Submission | Every resubmission must create a new version (v1, v2…). Old versions must never be deleted. Enables rollback, side-by-side comparison, and complete revision history for editorial disputes. |
| **BR-76** | Manuscript Lifecycle | *Structural* | **High** | P5 / P6 | Enforces the state machine: Draft → Submitted → Under Review → Revision Required or Approved. No step may be skipped. Prevents a manuscript from jumping to Approved without going through editorial review. |
| **BR-83** | Maximum Revision Limit | *Behavioral* | **High** | P6 — Editor Review | A chapter is capped at 3 consecutive revision rounds; thereafter the case escalates to the Board. Prevents indefinite revision loops that block publication schedules. |

| **BR-87** | **Vote Data Entry Authority** | *Behavioral* | **Critical** | P7 — Vote Data Entry | Only active Editorial Board members may enter VoteRecords. Prevents manipulation of ranking data by series creators who have an obvious incentive to inflate their own scores. |
| **BR-88** | **VoteRecord Uniqueness** | *Structural* | **Critical** | P7 — Vote Data Entry | Each series may have exactly one VoteRecord per period. Duplicate records would double-count votes, distorting rankings and triggering incorrect cancellation flags. |
| **BR-89** | **VoteRecord Validation** | *Definitional* | **Critical** | P7 — Vote Data Entry | A VoteRecord requires a valid period, readerCount ≥ voteCount, and both values ≥ 0\. Rejects impossible data (more votes than readers) before it reaches the ranking formula. |
| **BR-90** | **Ranking Formula** | *Calculational* | **Critical** | P8 — Ranking | Ranking score \= (voteCount / readerCount) × 100%; if readerCount \= 0 then score \= 0\. The canonical formula for the entire ranking system; handles the divide-by-zero edge case explicitly. |
| **BR-92** | **Automatic Ranking Recalculation** | *Behavioral* | **Critical** | P8 — Ranking | Rankings must be automatically recalculated whenever a VoteRecord is confirmed. Manual or delayed recalculation would mean displayed rankings lag behind actual vote data. |
| **BR-94** | Bottom 20% Review Flag | *Calculational* | **High** | P8 — Ranking | The bottom 20% of ranked Active series are automatically flagged for Cancellation Review. If fewer than 5 series are ranked, only a report is generated (no auto-flag). Drives the cancellation pipeline. |
| **BR-100** | Mandatory Cancellation Review | *Behavioral* | **High** | P8 — Ranking | A series in the bottom 20% for 3 consecutive periods enters Mandatory Cancellation Review. | **P9 — Cancel / Change Publication Decision** |  |  |  |  |  |
| **BR-101** | **Cancellation Decision Quorum** | *Structural* | **Critical** | P9 — Cancel Decision | Cancel or Change Publication Type decisions require ≥ 3 valid votes (same quorum rule as BR-29 but explicitly re-stated for P9). Ensures high-stakes decisions cannot be made by a single editor. |
| **BR-104** | **Cancellation Workflow Effect** | *Behavioral* | **Critical** | P9 — Cancel Decision | When a series is Cancelled: all In-Progress chapters and tasks are Suspended; data is archived. Provides a clean, consistent system state after cancellation and prevents orphaned active tasks. |
| **BR-105** | **Transaction Consistency** | *Structural* | **Critical** | P9 — Cancel Decision | Decision finalization and all downstream entity updates (status changes, archiving) must occur within a single database transaction. | **BR-106** | **Irreversible Cancellation Rule** | *Behavioral* | **Critical** | P9 — Cancel Decision | Cancelled series cannot be reactivated through the standard UI. Any reactivation requires Admin override with an audit log entry. Prevents accidental or unauthorized reversal of board decisions. |
| **BR-108** | **Cancellation Majority Rule** | *Calculational* | **Critical** | P9 — Cancel Decision | Cancellation requires quorum AND \> 50% of valid votes. Failure to reach majority yields a Continue outcome. Mirrors the series-approval majority rule and prevents cancellation by a slim or invalid vote. |

## Proposed New Rules (SU26-v1.1 — Series Proposal Extension)

These rules were identified during implementation of the proposal workflow and are not yet in the original BR catalog. They define behavior for tie and no-quorum board decisions that cannot be finalized by normal majority/quorum rules.

| ID | Rule Name | Type | Level | Flow | Description & System Impact |
| ----- | ----- | ----- | ----- | ----- | ----- |
| **BR-New-01** | **Editor-in-Chief Deadline Extension Rule** | *Behavioral* | **Critical** | P2 — Board Voting | When a proposal board decision reaches its first deadline and cannot be finalized because it is tied or has fewer than 3 valid votes, the Editor-in-Chief may extend the voting deadline exactly **once**. The extension must record `ExtensionCount = 1`, `ExtendedBy`, `ExtendedAt`, and `ExtensionReason` on the `BoardDecision`. A decision that has already been extended cannot be extended again. |
| **BR-New-02** | **Editor-in-Chief Special Decision Rule** | *Behavioral* | **Critical** | P2 — Board Voting | If an extended voting deadline passes and the board decision still cannot be finalized by normal quorum/majority rules, the Editor-in-Chief may make a special final decision of `Approved` or `Rejected`. The special decision must store `SpecialDecisionBy`, `SpecialDecisionAt`, and `SpecialDecisionReason`. If rejected, the reason must be written to `Series.RejectReason`. If approved, `Series.Status` is set to `Approved`. |
